## 1. 项目概述

### 1.1 愿景
构建一个**工业级、可扩展、自我进化**的 Code Agent，能够精准处理复杂的软件开发任务，同时保留处理通用任务的灵活性。该项目不是为了追逐单一模型的性能极限，而是通过卓越的**工程化架构 (Harness)** 来驾驭模型，确保输出质量、安全性和可靠性。

### 1.2 核心原则
- **工程确定性驾驭模型概率性**：核心流程由确定性中间件控制，模型只在沙箱内提出建议。
- **开闭原则**：核心闭环，能力开放。所有功能通过插件、Hook、协议扩展。
- **质量内建**：规约驱动 (SDD) 和测试驱动 (TDD) 不是可选功能，而是刻在 Agent 工作流 DNA 里的强制步骤。
- **持续进化**：能自动提炼任务经验为可复用技能，并持续评估自身性能。

## 2. 总体架构：七层参考模型 (ETCLOVG)

系统严格遵循 **ETCLOVG 七层模块化架构**，确保各模块独立迭代、职责清晰。

| 层级 | 名称 | 职责 | 关键实现 |
| :--- | :--- | :--- | :--- |
| **L1** | **治理层 (Governance)** | 模型生命周期管理、版本控制、策略执行 | 技能版本管理、AB 测试框架 |
| **L2** | **验证层 (Validation)** | 安全合规检查、输出格式验证、架构约束校验 | 确定性中间件、自定义 Linter |
| **L3** | **观测层 (Observation)** | 全链路日志、指标、追踪 | OpenTelemetry、结构化日志 |
| **L4** | **逻辑层 (Logic)** | 处理业务规则、TDD/SDD 工作流管线、质量保障集群 | 工作流引擎、技能库 |
| **L5** | **控制层 (Control)** | Agent 核心循环、任务调度、Hook 与中间件管理 | 事件驱动引擎 |
| **L6** | **工具层 (Tooling)** | 工具注册与发现、MCP 协议适配、沙箱管理 | MCP Server/Client、沙箱接口 |
| **L7** | **执行层 (Execution)** | 模型推理输入输出规范、不同 LLM 适配 | 模型路由、统一接口 |

## 3. 核心 Harness 引擎设计

### 3.1 事件驱动的插件化核心循环
Agent 的核心是 `感知→决策→执行` 循环，但完全由**事件总线**和**Hook**驱动，本身不包含任何业务逻辑。

```python
# 伪代码：极简但可扩展的 Agent Loop
class AgentLoop:
    def __init__(self, event_bus, hooks):
        self.event_bus = event_bus
        self.hooks = hooks

    async def run(self, task):
        await self.hooks.execute("before_perceive", task)
        context = await self.perceive(task)
        
        await self.hooks.execute("before_decide", context)
        decision = await self.decide(context)
        
        await self.hooks.execute("before_act", decision)
        result = await self.act(decision)
        
        await self.hooks.execute("after_act", result)
        return result
```

- **观察者模式**：UI、日志、监控等组件均通过订阅事件工作。
- **Hook 系统**：所有关键生命周期节点均预留 Hook（如 `before_llm_call`, `after_tool_execution`），外挂逻辑永不侵入核心代码。

### 3.2 控制流：确定性中间件与安全锚点
在关键路径上，**不信任模型输出**，强制执行确定性逻辑。

- **安全防火墙**：文件写入、网络请求、Shell 命令在工具执行层由确定性中间件再次校验权限。
- **流程锚点**：如`open_pr_if_needed`步骤，无论模型是否要求，只要满足条件（有提交且有通过测试），系统自动创建 PR。
- **架构约束 Linter**：将依赖规则、命名规范等编写为自定义 Linter，Agent 提交代码前自动运行，不通过则阻止提交。

## 4. 方法论的工程化集成

### 4.1 规约驱动开发 (SDD)
集成 `OpenSpec` 的理念，管理规约的完整生命周期。
- **规约管理**：`/spec` 命令创建 `specs/` 文件夹下的规格文件，使用结构化模板。
- **强制对齐**：任务开始时，Harness 强制将当前 `spec.md` 注入系统提示，并要求 Agent 输出符合规约的《执行计划》，经用户确认后方可编码。
- **归档与演进**：任务完成后，规约归档至 `specs/archive/`，变更被记录。

### 4.2 测试驱动开发 (TDD) 强制循环
核心工作流管线实现“红灯 → 绿灯 → 重构”的不可跳过闭环。

```yaml
# workflow/tdd_cycle.yaml
name: TDD Cycle
steps:
  - id: write_test
    skill: test_generator
    output: "failing test"
  - id: run_test_red
    skill: test_runner
    expect: "FAIL"
  - id: write_code
    skill: code_generator
    depends_on: [run_test_red]
  - id: run_test_green
    skill: test_runner
    expect: "PASS"
  - id: refactor
    skill: refactor_tool
    depends_on: [run_test_green]
  - id: final_validation
    skill: linter
    depends_on: [refactor]
```

- **Ralph 引擎模式**：借鉴 `MoAI-ADK`，存在一个监督 Agent 不断检查循环状态，确保不跳过失败步骤。

## 5. 智能体能力模块

### 5.1 分层提示词工程
提示词不再是字符串，而是按结构组装的**动态上下文块**。

| 块 | 内容 | 注入时机 | 管理方式 |
| :--- | :--- | :--- | :--- |
| **基础指令** | 角色、安全守则、输出格式 | 会话初始化 | 版本化模板文件 |
| **能力声明** | 当前可用工具列表（JSON Schema） | 每次决策前 | 从工具注册表动态渲染 |
| **项目约定** | `CONVENTIONS.md` | 会话初始化 | 从项目仓库读取 |
| **任务上下文** | 需求、规格、相关代码 | 任务开始 & 相关文件变化时 | 上下文检索管道 |
| **反馈上下文** | 测试报错、Lint 结果 | 工具执行后 | 结构化错误报告 |

### 5.2 上下文工程管道
构建智能检索与压缩管道，保障窗口内信息密度最大化。

- **分层检索**：`需求关键词提取 -> ripgrep 全文搜索 -> Tree-sitter AST 分析`，返回**最小相关代码块+其所属模块签名**。
- **地图注入**：始终注入一份可读的代码库地图（类似 `repomap`、`codmap`），标注文件大小和最近修改时间。
- **阶梯式摘要**：每 N 轮对话，用小型模型或 Agent 自身进行对话历史结构化摘要，替代原始消息。
- **状态锚点文件**：强制 Agent 读写 `claude-progress.txt`，记录 `[当前任务], [下一步], [操作哈希]`，作为断点续传的确定性锚点。

### 5.3 三层记忆架构

| 记忆类型 | 存储内容 | 生命周期 | 实现方式 |
| :--- | :--- | :--- | :--- |
| **工作记忆** | 当前任务变量、临时文件路径 | 单任务 | 结构化上下文对象 |
| **会话记忆** | 用户偏好、本轮对话决策 | 单会话 | 可持久化到 `session.json` |
| **长期记忆/技能** | 提炼后的通用技能、架构决策 | 跨会话 | `skills/` 文件库，Git 版本控制 |

- **技能提炼闭环（Nudge Engine）**：任务成功后，触发反思步骤 → Agent 总结模式并生成结构化技能文件 → 存入 `skills/` → 下一次任务，根据关键词自动检索并注入相关技能。
- **衰退与冲突**：记忆带时间戳和置信度，定期清理低权重项，冲突时询问用户。

## 6. 工具系统与集成

### 6.1 基于 MCP 协议的工具架构
- **标准内部总线**：所有工具（文件、Shell、Git、浏览器）均实现为 MCP Server。
- **生态即用**：可直接接入任何社区 MCP Server（如 Postgres、Slack），扩展零成本。
- **三层架构**：`Schema 定义`（自动生成 Function Call 描述）、`Registry`（自动发现与注册）、`Handler`（逻辑，含权限校验中间件）。

### 6.2 运行时注入与沙箱
- **资源注入**：数据库连接、当前用户 ID 等在运行时通过依赖注入容器提供，不污染工具的 Schema 定义。
- **分层沙箱**：控制层（Harness）与执行层（沙箱）物理分离。所有工具调用经过能力过滤（只读/需确认/自动执行）。

## 7. 多 Agent 协作系统

### 7.1 编排-执行者模式 (Orchestrator-Worker)
- **主 Agent (PM)**：负责任务分解、计划生成、进度追踪、冲突裁决。
- **子 Agent 集群**：
    - `Code Generator`：负责编码。
    - `Test Engineer`：负责编写测试和验证。
    - `Reviewer`：负责架构合规性检查。
    - `DevOps`：负责环境配置、CI 交互。
- **通信协议**：所有 Agent 通过统一 `TaskRequest`/`TaskResponse` 消息通信，基于消息队列 (NATS) 或 gRPC，天然支持跨语言、跨进程。

### 7.2 自主质量保障集群
多 Agent 组成质量流水线，自动化并行工作：
- **测试生成 Agent**：自动生成单元、集成、验收测试。
- **自我修复 Agent**：监控构建失败，自动尝试修复。
- **混沌工程 Agent**：主动注入故障，验证系统弹性。

## 8. 安全防御体系 (纵深防御)

| 防御层 | 策略 |
| :--- | :--- |
| **安全计算层** | 控制面（Harness）与数据面（沙箱）严格隔离，所有 I/O 经安全网关。 |
| **插件化安全系统** | 集成类似 `ClawAegis` 的专用安全插件，防御代码注入、恶意依赖。 |
| **全生命周期权限** | 精细化分层授权：只读、白名单命令、用户确认、自动执行。高风险操作需“双 Agent 互审”。 |
| **安全审计** | 所有 Agent 行为生成不可变审计日志。 |

## 9. 全链路可观测性

- **三大支柱**：集成 OpenTelemetry 标准，收集 Traces、Metrics、Logs。
- **Agent “感官”**：
    - **视觉**：浏览器自动化协议，获取 UI 截图、DOM 分析。
    - **诊断**：内置日志/指标查询工具，允许 Agent 自查运行时状态。
- **评估器 Agent**：独立的裁判 Agent，自动对任务完成度、代码质量、安全性进行多维度评分，输出 `SCORE.md`。

---

## 10. 长时任务与断点续传

- **Initializer + Coding Agent 模式**：先遣 Agent 搭建环境、生成待办列表（`tasks.md`）和进度日志；主力 Agent 每次只取一个小任务执行、提交、更新进度。
- **进度锚点强制读写**：每轮交互前后，系统强制 Agent 更新 `progress.json`，刷新上下文时依此恢复。
- **任务状态机**：定义明确的任务阶段（`INIT`, `PLAN`, `EXEC`, `TEST`, `REVIEW`, `DONE`），驱动状态转移，防止逻辑漂移。

---

## 11. 治理与持续进化

- **数据飞轮**：成功/失败案例被记录、分析、提炼为技能或约束，存入知识库。
- **AB 测试框架**：技能和提示词版本变更，通过 AB 测试验证效果后才全面上线。
- **结构化治理**：使用 `Block Merge` 规则定义架构约束，机器可读取并自动验证。
